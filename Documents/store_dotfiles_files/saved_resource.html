                                              Another problem I believe is easier to deal with with my approach is config files that are not in $HOME. For example if there is a file ~/.config/git/gitk, it takes precedence over ~/.gitk. Although it's not (necessarily) in $HOME, it makes sense to track it as a dot file anyway.

                                              I solved that by removing ~/.config/git/gitk (mv to gitk.backup, that is) within the script that creates links for a new user. Since I'm running the script anyway...

                                              Another way would have been to create the link ~/.config/git/gitk to the .gitk in my dotfiles repo.

                                              Still working with your out of repo work tree (since gitk falls back to ~/.gitk), but you need to remove ~/.config/git/gitk manually.

                                                      Gauthier Östervall hvtuananh • 6 months ago

                                                      Let's compare.
                                                      New machine: you need to 1. create your alias, 2. add to .gitignore, 3. clone, (3b. do you really need to redefine the alias?) 4. backup (maybe add a backup script to your repo?), 5. checkout, 6. unset showUntrackedFiles. In contrast, I need to: 1. git clone, 2. cd ~/dotfiles && ./setlinks

                                                      New file: you need to remember your special command `config`, other than that it's the regular way. I need to go to the repo and make the link before `git add`.

                                                      New changes: you need to remember to run with `config` instead of `git`. I need to cd to my repo and not remember anything special, it's vanilla git. Going to the repo before comitting changes is not something I'm likely to forget.

                                                      In my use case I very seldom add new files. I like to see all the files of my repo, which bare makes more cumbersome. I also value using vanilla git without hiding behind an alias, I'm more likely to remember how to do it. I understand your use case might be different.

                                                      Have a look, I just implemented my way, so I may encounter unexpected problems:
                                                      https://github.com/fleutot/...
                                                      I feel that this way is closer to the Unix philosophy. Git controls versions and pulls/pushes, it is not involved in what the files do or where they are. It's the local system that is in charge of having the files at the right place to be useful. Since having the repo at the right place (~) is a problem, I'd rather leave git out of this problem altogether. To each tool its responsibility.
